# Operator Intelligence Documentation Index

Version: commercial v1.0  
Folder alignment: `docs/`  
Status: Canonical documentation registry

## 1. Purpose

Register the root operating documentation for commercial v1.0 and identify where detailed authority resides.

## 2. Registry

| Document | Purpose |
|---|---|
| `methodology.md` | Canonical methodology summary, lifecycle, formulas, findings, routing, publication, and authorization |
| `commercial-v1-usage.md` | Practical paid-engagement operating sequence and required records |
| `commercial-v1-release-readiness.md` | Requirement inventory, folder gates, quantitative inventory, boundaries, and release recommendation |
| Root `README.md` | Product, architecture, inventory, usage, packages, examples, and release status |
| Root `ROADMAP.md` | Completed release path and governed post-v1 backlog |
| Root `CHANGELOG.md` | Version history and commercial v1.0 release changes |
| Root `COMMERCIAL_V1_COMPLETION.md` | Final repository release decision |

## 3. Authority hierarchy

```text
Root product/release documentation
→ Folder completion gates
→ Standards and scoring authorities
→ Framework indexes and finding libraries
→ Templates and playbooks
→ Examples, research, and asset specifications
```

When a summary conflicts with a more specific current authority, use the specific authority and reopen the affected summary/gate.

## 4. Usage

- Start with `README.md` for orientation.
- Use `commercial-v1-usage.md` to run an engagement.
- Use `methodology.md` for the method summary.
- Follow `templates/index.md` and `playbooks/index.md` for record/workflow selection.
- Use folder completion gates to determine downstream eligibility.
- Use `commercial-v1-release-readiness.md` and `COMMERCIAL_V1_COMPLETION.md` for release state.

## 5. Commercial v1.0 connection

This registry closes the documentation layer and prevents stale or duplicate operating instructions from becoming competing sources of truth.
